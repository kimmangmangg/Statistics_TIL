# 8.1. 데이터 분석의 전체 프로세스
>## 데이터 분석의 3단계<br>
1️⃣ **설계 단계**<br>
- 무엇을 하고자 하는지 정의
- 실무자와 분석가 간의 협의<br>

2️⃣ **분석 및 모델링 단계**<br>
- 데이터 준비, 추출, 가공, 검토, 분석, 모델링
- KDD분석, CRISP-DM, SEMMA 등을 활용하여 분석 스케줄링<br>

3️⃣ **구축 및 활용 단계**<br>
- 최종 선정된 분석 모델의 성과 측정
- 필요시 IT시스템 구축<br><br>

>## CRISP-DM 분석 방법론<br>
<img width="988" alt="Image" src="https://github.com/user-attachments/assets/1dd609a4-4f69-4350-950b-31fd2bbebda8" /><br>
- **비즈니스 이해** : 현재 비즈니스 상황 평가, 데이터 마이닝 및 프로젝트 목표 결정
- **데이터 이해** : 데이터 품질 확인 및 탐색을 통해 데이터 이해
- **데이터 준비** : 데이터 선택, 정제 및 통합
- **모델링** : 모델링 기법 선정 및 테스트 디자인 생성, 모델 생성 및 평가
- **평가** : 결과 평가, 프로세스 검토, 다음 단계 결정
- **배포** : 모니터링 및 유지 관리 계획, 배포 계획<br><br>

- **Sampling** : 분석용 데이터 추출(대표성을 가지는 표본 추출 등)
- **Exploration** : 통계치 확인, 변수 간 관계성 파악, 변수 선정, 데이터에  맞춰 비즈니스 아이디어 도출 및 분석 방향성 수정
- **Modification** : 결측치 처리, 로그변환, 구간화, 주성분 분석, 최종 변수 선정, 새로운 변수 생성
- **Modeling** : 비즈니스 목적에  맞는 모델 선정
- **Assessment** : 모델 예측력 등 성능 비교, 비즈니스 인사이트 적용, 추가 데이터 분석 진행<br><br>

# 8.2. 비즈니스 문제 정의와 분석 목적 도출
- 데이터 분석 프로젝트에 앞서 현재 문제점 파악 및 분석  목적 설정이 명확하지 못하면 제대로된 성과를 얻지 힘듦
- SCM의 Bullwhip effect같은 현상 발생 가능 !ㅠ<br>

### MECE(Mutually Exclusive Collectively Exhaustive)<br>
<img width="947" alt="Image" src="https://github.com/user-attachments/assets/aca49df7-f18f-4f33-b8ea-655fa42d4dfd" /><br>
- 비즈니스 문제를 올바르게 정의하기 위한 대표적인 논리적 접근법
- **겹치지 않게, 빠짐 없이 !**
- **로직 트리**를 통해 세부 항목을 정리함<br>
<img width="968" alt="Image" src="https://github.com/user-attachments/assets/a9e56cad-b701-4694-bbdb-f38cf22f71cb" /><br>
### 비즈니스 문제 정의<br>
- 현상에 대한 설명으로 끝나면 안되고, 본질적인 비즈니스 문제를 포괄해야 함<br><br>
**e.g.** <br> '통신사 약정이 끝난 고객이 타 통신사로 이탈한다.' = **현상에 대한 설명**<br> '~이탈하여 **회사의 수익이 감소하고 있다.**' = **비즈니스 문제**<br><br>
- 단순히 '~ 타 통신사로 이탈한다.'를 문제로 설정하면 '이탈 고객 예측 모델을 만드는 것'이 분석 목적이 되는 것이고
- '~ 타 통신사로 이탈하여 회사 수익이 감소하고 있다.'를 문제로 설정하면 '손해를 최소화 하는 프로모션 최적화 모델을 만드는 것'이 분석 목적이 되는 것 !
- 후자와 같이 문제 설정을 해야 분석을 통해 본질적인 해결방안이 나옴 !<br><br>

# 8.3. 분석 목적의 전환
- 기존의 분석 목적은 희미해지고, '이 데이터로 하라 수 있는 것이 무엇인가?'로 콘셉트가 바뀌는 경우, 그 순간을 확실히 인식하여 신속하게 수정/소통해 나가는 것이 중요 !
- 분석가와 실무자들 간의 협력(소통, 신뢰 형성 등)을 위해 데이터는 시각화해서 보여주는 것이 좋음<br><br>

# 8.4. 도메인 지식
- 해당되는 분야의 업에 대한 이해도
- 도메인 지식이 없는 상태에서 분석 진행시 1차원적일 수밖에 없음<br><br>
**e.g.<br>** 
배달 어플에 "국밥" 키워드 검색 노출 순위 최적화<br>
➡️ 요식업 도메인 : **지역별로 국밥이라는 단어를 말했을 때 떠올리는 음식이 다르다**<br>
➡️ 단순히 고객 군집화 후 인기 순으로 노출시키는 전략보다 더 심층적인 전략 제시 가능 !<br><br>

- 데이터가 생성되는 현장을 직접 보고, 소비자 혹은 사용자의 입장이 되어보는 경험이 도메인 지식을 얻는게 큰 도움을 줌 !<br><br>

# 8.5. 외부 데이터 수집과 크롤링
- 역시나 분석 목적을 명확히 한 후, 이에 맞는 외부 데이터를 수집해야 함
- 데이터를 구매하거나, 오픈 데이터를 수집하거나, 웹사이트를 크롤링하는 방법 존재<br>

### 크롤링<br>
- 프로그래미이을 통해 웹에 있는 데이터를 실시간으로 수집하는 방법
- robots.txt 파일에 접속 주체에 따라 크롤링 허용 범위를 안내<br><br>
<img width="943" alt="Image" src="https://github.com/user-attachments/assets/51b6f97c-95ce-4130-837c-d30bd2212966" /><br>
- 웹 페이지가 주어지면 그 페이지 내의 링크들을 따라가며 모든 내용을 다 가져오는 것<br>
  cf. 스크래핑 : 웹 페이지에서 자신이 원하는 부분의 정보만 가져오는 것
- Open API를 통해 정리된 데이터 제공 받거나, 코딩을 통해 원하는 데이터를 수집
- 파이썬 : BeautifulSoup , Selenium 등 사용
- 개발자 도구를 열어 웹사이트 HTML 구조를 확인하여 특정 위치에 해당되는 텍스트를 수집하는 방식 !

# 9.1. 데이터 분석 언어
<br>

### SAS(Statistical Analysis System)<br>
- 대표적인 제품형 데이터 분석 솔루션. 프로그래밍 언어보다는 솔루션에 가까움
- 역사가 깊은 상용 솔루션으로 신뢰도가 높음
- 가격은 높은 편이지만, 성능이 확실하여 정확성이 중요한 **금융업계**에서 많이 사용
- SAS Enterprise Miner라는 GUI기반 데이터 마이닝 도구 지원 ⭢ 프로그래밍 스킬이 부족한 사람들도 데이터 분석 가능!
- 기업 데이터 환경에 맞게 커스텀 받을 수 있음
- **데이터 시각화**를 쉽게 할 수 있다는 장점!
- 딥러닝, 인공신경망 분석에서는 R이나 파이썬보다 성능이 떨어짐<br>

### R<br>
- 오픈소스 데이터 분석용 언어
- 통계학자들을 염두에 두고 만들어졌기 때문에 **통계적 기능**이 우수, **데이터 시각화**에 특화
- ggplot2, ggvis, googleVis 사용 가능
- 활발한 커뮤니티를 통해 거의 모든 질문의 답 구할 수 있고, 지금까지 10만 개 이상의 데이터 분석 패키지가 공유됨
- 파이썬에 비해 프로그래밍적 소양이 부족해도 사용하는 데에 큰 무리가 없음<br>

### 파이썬(Python)<br>
- 오픈소스 데이터 분석용 언어
- C언어로 구현된 프로그래밍 언어로 데이터 분석 뿐만 아니라 웹서비스, 응용 프로그램, IoT 등 유연하게 사용 가능<br>
	cf. SAS와 R : 데이터 분석에 한정된 솔루션/언어<br>
- C언어보다 문법이 쉽고 간단하지만 10-350배 정도 느림
- 영상처리, 딥러닝, 고성능 게임 등은 C,C++ 사용
- 시각화 측면에서는 R에 비해 구현이 복잡한 편<br><br>

<img width="951" alt="Image" src="https://github.com/user-attachments/assets/89482aef-c4a4-4b6c-9302-c5a1fcdcd536" /><br><br>
